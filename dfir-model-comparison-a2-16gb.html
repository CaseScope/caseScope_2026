DFIR Model Selection: Ollama on A2 16GB

Evaluated for structured JSON, IOC extraction, event correlation, report generation, and chat copilot — your actual workloads from the codebase.

Recommendation

Qwen3 14B (Q4_K_M) is your best no-adapter option — strongest instruction following and structured output at 12GB VRAM, leaving room for KV cache. If you need fine-tuning, train a LoRA on Qwen3 8B and merge it — that's your safest adapter path on this hardware. Qwen3.5 9B is the fastest and smartest per-parameter, but has active Ollama tool-calling bugs — wait for v0.17.7+ stabilization before committing to it for production DFIR.

Detailed Analysis
Qwen3 14B ★ Recommended — No Adapter
Params: 14.8B dense VRAM: ~12GB Q4_K_M Context: 32K native, 131K YaRN License: Apache 2.0

The best balance of capability and VRAM fit for your A2. Widely benchmarked as the strongest instruction-following model in the 14B class. Excellent at structured JSON output with Ollama's native structured output mode. Think/no-think mode lets you toggle reasoning depth per-task. Stable tool calling in Ollama. 12GB footprint at Q4 leaves ~4GB for KV cache — enough for 8-16K context comfortably.

Qwen3 8B Best Adapter Target
Params: 8.2B dense VRAM: ~6GB Q4_K_M Context: 32K native, 131K YaRN License: Apache 2.0

If fine-tuning is the plan, this is your target. Dense architecture with the most mature LoRA/QLoRA pipeline in the Qwen family. Research confirms Qwen3 8B performed best among 8B models for DFIR IOC extraction in RAG setups (BERTScore 0.8338 in a published study). Unsloth, Axolotl, and standard HF training all support it cleanly. Merge→GGUF→Ollama is a well-trodden path. At ~6GB, it leaves 10GB for KV cache — plenty for your longer context needs in event correlation.

GPT-OSS 20B (GGUF) Viable — Tight Fit
Params: 21B MoE (3.6B active) VRAM: ~14GB MXFP4/GGUF Context: limited by VRAM License: Apache 2.0

You already know this model outperforms Qwen for your use case — 98.3% on structured output benchmarks. The fastest model on 16GB (140 t/s on RTX 4080). The problem: at ~14GB base weight, you have ~2GB for KV cache, limiting context severely. No native tool calling support in Ollama. The adapter merge→GGUF chain is the hard part (MXFP4 nn.Parameter format complicates conversion). If you can get a clean merged GGUF, it's excellent for batch processing tasks that don't need long context or tool calling.

The most battle-tested adapter pipeline in the ecosystem. If your priority is "I need fine-tuning to work reliably, period," this is the safest bet. Every training framework supports it perfectly. Good for the chat copilot path if fine-tuned on your DFIR domain.

Qwen3 30B-A3B Caution — VRAM Spill
Params: 30B MoE (3B active) VRAM: ~18GB Q4_K_M Context: 32K native License: Apache 2.0

Outcompetes QwQ-32B with only 3B active params — remarkable efficiency. But at ~18GB Q4, it exceeds your 16GB VRAM and forces CPU offloading. Expect 3-10x slowdown (roughly 10-20 t/s instead of 60+). Only worth considering for batch/offline jobs where latency doesn't matter. Not viable for interactive chat copilot.

Practical Strategy
Two-Model Approach

Run two models on the A2 (not simultaneously). Use Qwen3 14B for interactive tasks (chat copilot, report writing, correlation analysis) where instruction following and tool calling matter most. Use Qwen3 8B fine-tuned (or GPT-OSS 20B GGUF if the merge chain validates) for batch tasks (IOC extraction, checkpoint summaries, event summaries) where speed and strict JSON adherence matter more than conversational ability. Ollama makes model switching trivial — just change the model name in your API call.

If You Must Fine-Tune

Train on the A100 with Qwen3 8B + QLoRA via Unsloth. Build your training set from your actual DFIR pipeline outputs — the structured JSON from ioc_contract.py, the correlation outputs, the checkpoint summaries. Merge the adapter. Convert to GGUF Q5_K_M. Deploy to A2 via Ollama. Test against your real tasks before trusting it. If Qwen3 14B's zero-shot performance is close enough with good system prompts, skip fine-tuning entirely — it's an ongoing maintenance burden.
