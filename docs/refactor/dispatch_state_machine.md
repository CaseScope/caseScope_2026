## Purpose
Define the CaseScope tool-dispatch state machine, permission tiers, and provenance handling so the chat and subagent runtime do not get reinvented under different names later.

## Locked Enums

### ToolTier
These names are locked from Session B:
- `READ_SAFE`
- `READ_SENSITIVE`
- `WRITE_REVERSIBLE`
- `WRITE_COMMITTING`

Semantics:
- `READ_SAFE`: auto-allow
- `READ_SENSITIVE`: ask, cacheable per tool and case and session
- `WRITE_REVERSIBLE`: ask, cacheable per tool and case and session
- `WRITE_COMMITTING`: ask every time, never cacheable

### Provenance
These names are locked from Session C:
- `ANALYST`
- `SYSTEM_DERIVED`
- `ARTIFACT_TAINTED`
- `ELEVATED_RISK`
- `MODEL_SYNTHESIZED`

Semantics:
- `ANALYST`: typed by the human analyst
- `SYSTEM_DERIVED`: structural output from constrained artifact fields
- `ARTIFACT_TAINTED`: attacker-reachable or parser-unstable content
- `ELEVATED_RISK`: browser, Plaso, base64-decoded, or otherwise especially risky artifact-derived content
- `MODEL_SYNTHESIZED`: content generated by the model itself

Composition rule:
- provenance is monotonic under composition
- if an argument is built from multiple sources, the highest-risk provenance wins

## Locked State Machine
- `L0`: contextual gates
- `L1`: schema and provenance validation
- `L2`: permission or rule match
- `L3`: analyst interaction and execution

Flow:
1. model proposes tool use
2. `L0` checks:
   - evidentiary lock
   - case-scope isolation
   - license availability
3. `L1` checks:
   - required argument schema
   - provenance tagging
   - artifact-derived injection or misuse checks
4. `L2` checks:
   - `ToolTier`
   - cached allow or deny decision for cacheable tiers
5. `L3`:
   - analyst confirmation if required
   - execution
   - conversion to `tool_result`

Critical invariant:
- every tool call path must return a `tool_result`
- dispatch must not leave orphaned tool uses

## L0 Rules
- `case_id` must be explicit for case-scoped tool calls
- do not silently inherit missing case identifiers from context
- state-changing tools must fail under evidentiary lock
- licensed features must return a structured feature-unavailable result, not a silent miss

This was the repeated security lesson surfaced in Session C: silent defaults are a cross-case vulnerability class.

## L1 Rules

### Schema
- every tool input must validate against a strict schema before permission checks
- missing case-scoped fields must fail fast

### Provenance
Parser-tier policy from Session C:
- Tier 1 and Tier 5 structured artifacts:
  - constrained structural fields may be `SYSTEM_DERIVED`
  - attacker-controlled fields are `ARTIFACT_TAINTED`
- Tier 2 and Tier 3 semi-structured or unstable artifacts:
  - record-level content defaults to `ARTIFACT_TAINTED`
- Tier 4 browser and similar high-risk artifacts:
  - content defaults to `ELEVATED_RISK`

Current implementation note:
- parser-emitted provenance metadata now lands in stored event `extra_fields` from the shared `ParsedEvent` boundary
- producer surfaces such as forensic chat retrieval and chat tools now merge parser-emitted tags before dispatch consumption, so L1 validation can prefer emitted provenance over fallback defaults on those paths

### Injection Discipline
- artifact-derived content must not be treated as instructions
- cross-case instructions embedded in artifacts are prompt injection and must be ignored
- browser and decoded blob content receive stricter treatment than constrained system fields

## L2 Rules
- `READ_SAFE`: allow
- `READ_SENSITIVE`: ask unless cached
- `WRITE_REVERSIBLE`: ask unless cached
- `WRITE_COMMITTING`: always ask, never cache

Cache scope:
- `(tool_name, case_id, session_id)`

Never global:
- no cache decision may cross case boundaries
- no cached allow may override a deny

## L3 Rules
- analyst approval may be:
  - one-time allow
  - session-scoped allow for cacheable tiers
  - reject
  - reject and do not ask again for cacheable tiers
- rejections are represented as structured tool results, not thrown exceptions

## Locked Response Categories
These categories should exist as explicit response constructors rather than ad hoc strings:
- interrupt
- cancel
- reject
- reject with analyst reason
- auto reject by policy
- do-not-ask reject
- evidentiary lock
- cross-case denial
- feature unavailable
- subagent reject
